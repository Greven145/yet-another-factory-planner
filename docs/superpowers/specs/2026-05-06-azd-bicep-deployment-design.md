# Deployment Design: AZD + Bicep for Yet Another Factory Planner

**Date:** 2026-05-06  
**Status:** Approved  

## Goals

- Infrastructure as code checked into git
- Full control over provisioning (including resources Aspire doesn't model)
- Automated CD via GitHub Actions
- Serve frontend at `yafp.game.gottselig.ca`

## Out of Scope

- Key Vault / secrets management (no secrets beyond Cosmos DB connection string, which Aspire handles)
- VNet / private networking
- CDN / caching layer
- Multi-environment promotion (single environment for now)

---

## Architecture

```
GitHub Actions (azure-dev.yml)
  └── azd provision    → Bicep → Azure resources
  └── azd deploy
        ├── api        → build .NET container → push to ACR → deploy to ACA
        └── web        → upload dist/ → Azure Static Web Apps
```

**Azure resources:**

| Resource | SKU | Notes |
|---|---|---|
| Azure Container Apps Environment | Consumption | Hosts the API |
| Azure Cosmos DB | Serverless | Shared factory storage; 7-day TTL |
| Azure Static Web Apps | Free | React/Vite SPA frontend |
| CNAME record (`yafp` in `game.gottselig.ca`) | — | Updated in `dedicated` RG; points to SWA hostname |
| SWA custom domain | — | `yafp.game.gottselig.ca` via `cname-delegation` |

## AZD Service Definitions (`azure.yaml`)

```yaml
name: yet-another-factory-planner
services:
  api:
    project: ./api.web/api.web.csproj
    language: csharp
    host: containerapp
  web:
    project: ./client
    dist: dist
    language: js
    host: staticwebapp
```

AZD locates each Azure resource by the `azd-service-name` tag set in Bicep.

## Bicep Module Design

### Deployment order (enforced by `dependsOn` in `main.bicep`)

```
1. api.bicep        → ACA environment, ACR, Cosmos DB
2. swa.bicep        → Static Web Apps resource
                      outputs: defaultHostname, name
3. dns.bicep        → cross-RG (scope: dedicated)
                      updates yafp CNAME: yafp.azureedge.net → <swa>.azurestaticapps.net
4. swa-domain.bicep → customDomains child resource
                      validationMethod: cname-delegation
                      dependsOn: dns module
```

### `main.bicep` module wiring (abbreviated)

```bicep
module swa 'app/swa.bicep' = {
  name: 'swa'
  params: { location: location, appName: swaName }
}

module dns 'app/dns.bicep' = {
  name: 'dns'
  scope: resourceGroup('dedicated')
  params: { swaHostname: swa.outputs.defaultHostname }
}

module swaDomain 'app/swa-domain.bicep' = {
  name: 'swa-domain'
  dependsOn: [dns]
  params: {
    swaName: swa.outputs.name
    customDomain: 'yafp.game.gottselig.ca'
  }
}
```

### `swa.bicep`

- Resource: `Microsoft.Web/staticSites@2023-12-01`
- SKU: `Free`
- `provider: 'Custom'` (no GitHub-integrated CI; deployments via AZD)
- Tag: `azd-service-name: web`
- Outputs: `defaultHostname`, `name`

### `dns.bicep`

- Scope: `resourceGroup('dedicated')`
- References existing zone `game.gottselig.ca` with `existing` keyword
- Updates (not creates) the `yafp` CNAME record: target → `swaHostname` parameter
- TTL: 3600 (preserve existing)

### `swa-domain.bicep`

- Resource: `Microsoft.Web/staticSites/customDomains@2023-12-01`
- `name: 'yafp.game.gottselig.ca'`
- `validationMethod: 'cname-delegation'`
- Requires CNAME to already resolve to SWA hostname (guaranteed by `dependsOn: [dns]`)

## CI/CD: GitHub Actions

Two workflows:

| Workflow | Trigger | Purpose |
|---|---|---|
| `ci.yml` (existing) | PR + push | Build, test, lint — no deploy |
| `azure-dev.yml` (new, azd-generated) | Push to `main`/`master` | Provision + deploy |

`azure-dev.yml` generated by `azd pipeline config`. Uses **OIDC federation** (no long-lived service principal secret). Steps:

```
1. azd provision --no-prompt   # idempotent; only changes infra if Bicep changed
2. azd deploy --no-prompt      # deploys api (container) + web (SPA) in parallel
```

`azd deploy web` and `azd deploy api` can be run independently to decouple SPA and API deployments.

---

## Permissions Required

The AZD service principal (OIDC) needs:

| Resource Group | Role | Reason |
|---|---|---|
| App resource group | `Contributor` | Create/manage ACA, Cosmos, SWA, ACR |
| `dedicated` | `DNS Zone Contributor` | Update `yafp` CNAME in `game.gottselig.ca` |

`DNS Zone Contributor` is scoped to the zone level to avoid over-permissioning the `dedicated` RG.

## Local Development

No change. Aspire AppHost continues to orchestrate local dev:
- API runs via Aspire
- Cosmos DB runs via emulator (`RunAsPreviewEmulator()`)
- React dev server runs via Vite (`npm run dev`)

`azd` is only used for provisioning and deploying to Azure.

## Open Questions / Future Work

- Does the Free SWA SKU meet all requirements? (If enterprise-grade CDN or private networking is needed later, upgrade to Standard.)
