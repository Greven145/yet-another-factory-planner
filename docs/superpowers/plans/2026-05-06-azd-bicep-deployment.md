# AZD + Bicep Deployment Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Wire up a fully automated, IaC-in-git deployment pipeline using AZD + Bicep that provisions Azure Container Apps (API), Azure Cosmos DB, Azure Static Web Apps (frontend), and a DNS CNAME update, then deploys both services via GitHub Actions on push to main/master.

**Architecture:** AZD synthesizes Bicep from the Aspire AppHost manifest into `infra/`, giving us a git-tracked, customizable infrastructure definition. Three hand-written Bicep modules extend the generated baseline: one for Azure Static Web Apps, one for a cross-resource-group DNS CNAME update in the pre-existing `game.gottselig.ca` zone, and one for the SWA custom domain binding. GitHub Actions uses OIDC federation (no long-lived secrets) to run `azd provision` + `azd deploy` on every push.

**Tech Stack:** AZD CLI, Bicep, .NET Aspire 13.2.4, Azure Container Apps, Azure Cosmos DB, Azure Static Web Apps, GitHub Actions OIDC

## Task 3: Create `infra/app/swa.bicep`

**Files:**
- Create: `infra/app/swa.bicep`

- [ ] **Step 1: Create the file**

  Create `infra/app/swa.bicep` with this exact content:

  ```bicep
  param location string
  param appName string
  param tags object = {}

  resource staticWebApp 'Microsoft.Web/staticSites@2023-12-01' = {
    name: appName
    location: location
    sku: {
      name: 'Free'
      tier: 'Free'
    }
    properties: {
      provider: 'Custom'
      stagingEnvironmentPolicy: 'Enabled'
      allowConfigFileUpdates: true
    }
    tags: union(tags, {
      'azd-service-name': 'web'
    })
  }

  output defaultHostname string = staticWebApp.properties.defaultHostname
  output name string = staticWebApp.name
  output id string = staticWebApp.id
  ```

  `provider: 'Custom'` disables Aspire's GitHub-integrated CI for SWA — deployments go through AZD instead. `azd-service-name: web` is the tag AZD uses to locate the SWA resource when running `azd deploy`.

## Task 4: Create `infra/app/dns.bicep`

**Files:**
- Create: `infra/app/dns.bicep`

This module is deployed with `scope: resourceGroup('dedicated')` from `main.bicep`. It updates the pre-existing `yafp` CNAME record in the `game.gottselig.ca` zone (currently pointing to the deprecated `yafp.azureedge.net`).

- [ ] **Step 1: Create the file**

  Create `infra/app/dns.bicep` with this exact content:

  ```bicep
  param swaHostname string

  resource dnsZone 'Microsoft.Network/dnsZones@2018-05-01' existing = {
    name: 'game.gottselig.ca'
  }

  resource yafpCname 'Microsoft.Network/dnsZones/CNAME@2018-05-01' = {
    parent: dnsZone
    name: 'yafp'
    properties: {
      TTL: 3600
      CNAMERecord: {
        cname: swaHostname
      }
    }
  }
  ```

  The `existing` keyword references the DNS zone without managing its lifecycle — it won't delete the zone if the module is removed. Bicep will update the `yafp` record in place.

- [ ] **Step 2: Validate**

  ```powershell
  az bicep build --file infra/app/dns.bicep
  ```

  Expected: no errors.

- [ ] **Step 3: Commit**

## Task 5: Create `infra/app/swa-domain.bicep`

**Files:**
- Create: `infra/app/swa-domain.bicep`

This module must run AFTER the DNS update (Task 4) because SWA validates the custom domain by confirming the CNAME points to the SWA hostname. The `dependsOn` in `main.bicep` (Task 6) enforces this order.

- [ ] **Step 1: Create the file**

  Create `infra/app/swa-domain.bicep` with this exact content (use `2024-11-01` to match `swa.bicep`):

  ```bicep
  param swaName string

  resource staticWebApp 'Microsoft.Web/staticSites@2024-11-01' existing = {
    name: swaName
  }

  resource customDomain 'Microsoft.Web/staticSites/customDomains@2024-11-01' = {
    parent: staticWebApp
    name: 'yafp.game.gottselig.ca'
    properties: {
      validationMethod: 'cname-delegation'
    }
  }
  ```

  `cname-delegation` validation works for subdomains (not apex). It succeeds once the CNAME at `yafp.game.gottselig.ca` resolves to the SWA default hostname — which `dns.bicep` ensures.

## Task 6: Add SWA modules to `infra/main.bicep`

**Files:**
- Modify: `infra/main.bicep`

- [ ] **Step 1: Find where to insert the SWA module**

  Open `infra/main.bicep`. Locate the last `module` block (the last one before any `output` declarations). The new SWA module goes after it.

- [ ] **Step 2: Add the SWA module**

  Insert after the last existing module block, before any `output` lines:

  ```bicep
  module swa './app/swa.bicep' = {
    name: 'swa'
    scope: rg
    params: {
      location: location
      appName: 'stapp-${resourceToken}'
      tags: tags
    }
  }
  ```

  Note: `rg` is the resource group resource declared in the generated `main.bicep` (typically `resource rg 'Microsoft.Resources/resourceGroups@2022-09-01' = { ... }`). Use whatever variable name the generated file uses. `resourceToken` is the generated unique suffix variable. If `abbreviations.json` contains a key for `webStaticSites`, you can use `'${abbrs.webStaticSites}${resourceToken}'` instead of the hardcoded `'stapp-'` prefix.

- [ ] **Step 3: Add the DNS module**

  Insert after the SWA module:

  ```bicep
  module dns './app/dns.bicep' = {
    name: 'dns'
    scope: resourceGroup('dedicated')
    params: {
      swaHostname: swa.outputs.defaultHostname
    }
  }
  ```

  This uses `scope: resourceGroup('dedicated')` to deploy into the `dedicated` resource group where the `game.gottselig.ca` DNS zone lives. This works because `main.bicep` has `targetScope = 'subscription'`.

  If `main.bicep` is NOT subscription-scoped (verify with `targetScope` at the top of the file), replace `resourceGroup('dedicated')` with `resourceGroup(subscription().subscriptionId, 'dedicated')`.

- [ ] **Step 4: Add the SWA domain module**

  Insert after the DNS module:

  ```bicep
  module swaDomain './app/swa-domain.bicep' = {
    name: 'swa-domain'
    scope: rg
    dependsOn: [dns]
    params: {
      swaName: swa.outputs.name
    }
  }
  ```

  `dependsOn: [dns]` ensures the CNAME update completes before SWA attempts domain validation.

- [ ] **Step 5: Add outputs**

  In the `output` section at the bottom of `main.bicep`, add:

  ```bicep
  output AZURE_STATIC_WEB_APP_NAME string = swa.outputs.name
  output AZURE_STATIC_WEB_APP_HOSTNAME string = swa.outputs.defaultHostname
  ```

  AZD reads `UPPER_SNAKE_CASE` outputs and writes them to `.azure/<env>/.env`, making them available to hooks and the `azd deploy` step.

- [ ] **Step 6: Validate the full template**

  ```powershell
  az bicep build --file infra/main.bicep
  ```

  Expected: no errors.

- [ ] **Step 7: Commit**

  ```powershell

  git commit -m "feat: wire swa, dns, and custom domain modules into main bicep"
  ```

## Task 7: Update `azure.yaml` for the web service

**Files:**
- Modify: `azure.yaml`

The Vite client reads `VITE_REACT_APP_API_BASE_URL` at **build time** (`import.meta.env.VITE_REACT_APP_API_BASE_URL` in `client/src/api/index.ts`). There is no `.env.production` file — without one, the production build would use `undefined` as the API base URL. We fix this with a `predeploy` hook on the `web` service that writes `.env.production` from the AZD environment after the `api` service has deployed.

Because the hook reads `SERVICE_API_ENDPOINT_URL` (set only after the `api` service is deployed), **services must deploy sequentially: api first, then web**. The GitHub Actions workflow is updated in Task 9 to enforce this.

- [ ] **Step 1: Add `.env.production` to `.gitignore`**

  Open `.gitignore` and add:
  ```
  client/.env.production
  ```

  This file is generated at deploy time by the hook — it must not be committed.

- [ ] **Step 2: Replace the generated `azure.yaml`**

  Replace the entire content of `azure.yaml` with:

  ```yaml
  # yaml-language-server: $schema=https://raw.githubusercontent.com/Azure/azure-dev/main/schemas/v1.0/azure.yaml.json

  name: yet-another-factory-planner
  services:
    api:
      project: ./YetAnotherFactoryPlanner.AppHost/YetAnotherFactoryPlanner.AppHost.csproj
      language: dotnet
      host: containerapp
    web:
      project: ./client
      dist: dist
      language: js
      host: staticwebapp
      hooks:
        predeploy:
          windows:
            shell: pwsh
            run: |
              $apiUrl = azd env get-value SERVICE_API_ENDPOINT_URL
              Set-Content -Path .env.production -Value "VITE_REACT_APP_API_BASE_URL=$apiUrl"
          posix:
            shell: sh
            run: |
              echo "VITE_REACT_APP_API_BASE_URL=$(azd env get-value SERVICE_API_ENDPOINT_URL)" > .env.production
  ```

  The `api` service points to the **AppHost** (not `api.web` directly) — AZD uses Aspire to resolve the actual container image. The `predeploy` hook on `web` writes `client/.env.production` before AZD runs `npm run build`, injecting the live API URL into the Vite bundle.

- [ ] **Step 3: Verify the schema**

  ```powershell
  azd config show
  ```

  Expected: no validation errors. (AZD validates `azure.yaml` format.)

- [ ] **Step 4: Commit**

  ```powershell
  git add azure.yaml .gitignore
  git commit -m "feat: update azure.yaml with api and web service definitions and predeploy hook"
  ```

## Task 10: Assign DNS Zone Contributor permission

**Files:** none (Azure RBAC, not code)

The service principal created by `azd pipeline config` only has `Contributor` on the app resource group. It needs `DNS Zone Contributor` on the `game.gottselig.ca` zone to update the `yafp` CNAME via `dns.bicep`.

- [ ] **Step 1: Get the service principal object ID**

  ```powershell
  $clientId = azd env get-value AZURE_CLIENT_ID
  $spObjectId = az ad sp show --id $clientId --query id -o tsv
  Write-Host "SP Object ID: $spObjectId"
  ```

- [ ] **Step 2: Get the DNS zone resource ID**

  ```powershell
  $zoneId = az network dns zone show `
    --name game.gottselig.ca `
    --resource-group dedicated `
    --query id -o tsv
  Write-Host "Zone ID: $zoneId"
  ```

- [ ] **Step 3: Assign the role**

  ```powershell
  az role assignment create `
    --role "DNS Zone Contributor" `
    --assignee-object-id $spObjectId `
    --assignee-principal-type ServicePrincipal `
    --scope $zoneId
  ```

  Expected output: JSON object with `roleDefinitionId` containing the DNS Zone Contributor GUID and `principalId` matching `$spObjectId`.

- [ ] **Step 4: Verify**

  ```powershell
  az role assignment list `
    --assignee $spObjectId `
    --scope $zoneId `
    --query "[].{role:roleDefinitionName}" -o table
  ```

  Expected: `DNS Zone Contributor` in the output.
