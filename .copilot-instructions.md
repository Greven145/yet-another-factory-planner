# Copilot Instructions

## Architecture

This is a **Satisfactory factory planner** web app. The client is a React 19 SPA that performs all production-chain calculations in-browser using a GLPK.js linear programming solver. The .NET API is minimal — it only serves game data and persists shared factories to CosmosDB.

### Stack
- **`client/`** — React 19 + TypeScript (strict mode), built with Craco (CRA override). Styling via styled-components + Mantine v8.
- **`api/`** — .NET 10 Azure Functions v4 (isolated worker model). Three endpoints: `GET /ping`, `GET /initialize`, `POST /share-factory`.
- **`ParseDocs/`** — .NET CLI tool that parses Satisfactory docs JSON into game data resource files embedded in the API.
- **`YetAnotherFactoryPlanner.AppHost/`** — .NET Aspire 13 orchestration for local development.
- **`YetAnotherFactoryPlanner.ServiceDefaults/`** — Shared Aspire configuration for observability.

### Data Storage
- **Cosmos DB** for shared factory configurations
- **Local Development**: Linux-based Cosmos DB emulator (vnext-preview) running in Docker via Aspire

## Running the Application

### Local Development with Aspire

The project uses **.NET Aspire 13** for local development orchestration. This provides a unified development experience with built-in observability and service discovery.

**Prerequisites:**
- .NET 10 SDK
- Node.js (for React client)
- Docker Desktop (for Cosmos DB emulator)
- Aspire CLI (installed globally)

**Start everything:**
```bash
dotnet run --project YetAnotherFactoryPlanner.AppHost
```

This single command will:
- Start the Cosmos DB Linux emulator (with HTTPS enabled)
- Start the Azure Functions API (port 7071)
- Start the React development server (port 3000)
- Launch the Aspire Dashboard for monitoring

**Service URLs when running:**
- React Client: http://localhost:3000
- Azure Functions API: http://localhost:7071
- Cosmos DB Emulator: https://localhost:8081
- Cosmos Data Explorer: http://localhost:1234
- Aspire Dashboard: Opens automatically (typically http://localhost:15XXX)

### Production Deployment

For containerized deployments and CI/CD, docker-compose is still available:
```bash
docker-compose up
```

## Aspire MCP Server Integration

This project has an active Aspire MCP server connection that provides tools to inspect and manage the Aspire application:

### Available Aspire Tools

1. **aspire-list_resources** - List all resources (projects, containers, endpoints)
2. **aspire-list_console_logs** - Get console logs for a specific resource
3. **aspire-execute_resource_command** - Execute commands on resources (start, stop, restart)
4. **aspire-list_structured_logs** - Get structured logs for analysis
5. **aspire-list_traces** - View distributed traces across services
6. **aspire-list_trace_structured_logs** - Get logs for a specific trace

### Common Aspire Workflows

**Check if everything is running:**
```
Use aspire-list_resources to see all services and their status
```

**Debug a service that's not working:**
```
1. Use aspire-list_resources to identify the resource name
2. Use aspire-list_console_logs with the resource name to see its output
3. Use aspire-list_structured_logs to see detailed logging
```

**Investigate an error in distributed tracing:**
```
1. Use aspire-list_traces to find traces with errors
2. Use aspire-list_trace_structured_logs with the trace ID to see detailed logs
```

**Restart a service:**
```
1. Use aspire-list_resources to get the resource name
2. Use aspire-execute_resource_command with command "resource-restart"
```

### Resource Names
- `cosmosdb` - Cosmos DB emulator container
- `api` - Azure Functions project
- `client` - React npm application

### When to Use Aspire Tools

- **Debugging**: When services aren't starting or have errors
- **Monitoring**: To check health status and logs
- **Development**: To understand how services are communicating
- **Troubleshooting**: When API calls fail or data isn't flowing correctly

**Example troubleshooting flow:**
If the API can't connect to Cosmos DB:
1. Check if cosmosdb resource is running: `aspire-list_resources`
2. View Cosmos DB logs: `aspire-list_console_logs` with resourceName="cosmosdb"
3. View API logs: `aspire-list_console_logs` with resourceName="api"
4. Check structured logs for connection errors: `aspire-list_structured_logs` with resourceName="api"

## Troubleshooting

### Services won't start in Aspire
- Check Docker is running: `docker ps`
- View resource status: Use `aspire-list_resources` tool
- Check logs: Use `aspire-list_console_logs` tool with resource name

### npm install fails
Use `--legacy-peer-deps` flag due to TypeScript version conflicts:
```bash
npm install --legacy-peer-deps
```

### Cosmos DB connection errors
- Verify emulator is running in Aspire Dashboard
- Check certificate handling in `api/Program.cs`
- Verify connection string injection via Aspire tools

### Port conflicts
Default ports used: 3000 (React), 7071 (Functions), 8081 (Cosmos), 1234 (Data Explorer)
- Stop conflicting services or modify ports in `AppHost.cs`

## Testing Approach
